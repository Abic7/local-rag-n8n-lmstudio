WHAT IS RAG?
────────────────────────────────────────────────────────────────────────────
Retrieval-Augmented Generation = Retrieve relevant docs + feed to LLM
Result: Answers grounded in actual sources with citations


THE 5-STAGE PIPELINE
────────────────────────────────────────────────────────────────────────────

STAGE 1: DOCUMENT INGESTION
  Input:   PDF/Text files
  Process: Extract content + metadata
  Output:  Raw text with source tracking
  Node:    HTTP Request or File Read

STAGE 2: CHUNKING
  Input:   Raw text
  Process: Split into 512-token chunks with 50-token overlap
  Output:  Semantically coherent fragments
  Node:    Code node (JavaScript)

  Why overlapping?
  - Prevents info split across chunks
  - Maintains semantic context
  - Improves retrieval quality

STAGE 3: EMBEDDING
  Input:   Text chunks
  Process: Convert to dense vectors (1536-dim)
  Output:  Semantic vectors stored with metadata
  Node:    OpenAI text-embedding-3-small OR Ollama nomic-embed-text

  Model Choice:
  - OpenAI small: Fast, accurate, $0.02/1M tokens
  - Ollama (local): Free, no internet needed

STAGE 4: VECTOR STORAGE
  Input:   Embeddings + metadata
  Process: Index in searchable database
  Output:  Queryable vector index
  Node:    Qdrant (local) OR Pinecone (managed)

  Comparison:
  ┌─────────────┬──────────────┬──────────────┐
  │             │ Qdrant       │ Pinecone     │
  ├─────────────┼──────────────┼──────────────┤
  │ Setup       │ Docker       │ SaaS account │
  │ Cost        │ Free         │ $180/month   │
  │ Ops         │ High         │ None         │
  │ Latency     │ 50-100ms     │ 200-500ms    │
  └─────────────┴──────────────┴──────────────┘

STAGE 5: RETRIEVAL & GENERATION
  Input:   User question
  Process:
    1. Embed question (same model as Stage 3)
    2. Cosine similarity search (find top-5 similar chunks)
    3. Build RAG prompt with context
    4. Call LLM (Claude/GPT-4) with context
    5. Return grounded answer + citations
  Output:  Answer + [Source 1], [Source 2], etc.
  Nodes:   Embed → Retrieve → Build Prompt → LLM → Format Response


PERFORMANCE TARGETS
────────────────────────────────────────────────────────────────────────────

Embedding:        200-500ms
Vector Search:     50-150ms
LLM Generation:  1000-3000ms  ← Usually the bottleneck
──────────────────────────────
TOTAL:           2000-4000ms (acceptable for most use cases)

To improve:
  - Use faster LLM (gpt-4o-mini vs. gpt-4o)
  - Add Redis cache (for repeated questions)
  - Batch embed (process 100 at once)


KEY DESIGN DECISIONS
────────────────────────────────────────────────────────────────────────────

1. OVERLAPPING CHUNKS (50-token overlap)
   Without: "System has three components: [split] ...A, B, C"
   With:    "System has three...components: A, B, C..." (context preserved)

2. LOW TEMPERATURE (0.3)
   High temp (0.8):  Creative but hallucinates
   Low temp (0.3):   Grounded in context, less creative (GOOD FOR RAG)

3. SIMILARITY THRESHOLD (>0.75)
   <0.5: Probably not relevant
   0.5-0.75: Maybe relevant
   >0.75: Highly confident (use this)

4. EXPLICIT CITATIONS
   System prompt: "Always use [Source N] format"
   User prompt:   "Answer must cite sources"
   Result:        Traceable, auditable answers


FAILURE MODES & FIXES
────────────────────────────────────────────────────────────────────────────

PROBLEM: No relevant chunks retrieved (similarity <0.5)
ROOT CAUSE: Query and documents are too different
FIXES:
  1. Check embedding model consistency (same for query + training?)
  2. Review chunk quality (is the document relevant?)
  3. Lower similarity threshold (0.75 → 0.5)
  4. Add more relevant documents

PROBLEM: Hallucination (answer contradicts sources)
ROOT CAUSE: High temperature or ambiguous context
FIXES:
  1. Lower temperature (0.3 → 0.1)
  2. Add validation layer (check citations match content)
  3. Add explicit instruction: "Use ONLY provided sources"
  4. Smaller, more focused chunks

PROBLEM: Slow performance (>5 seconds)
ROOT CAUSE: Usually LLM generation (2-3 seconds is normal)
FIXES:
  1. Use faster LLM (gpt-4o-mini instead of gpt-4o)
  2. Add Redis cache (30-40% hit rate typical)
  3. Use regional API endpoints
  4. Batch process multiple queries


INTERVIEW TALKING POINTS
────────────────────────────────────────────────────────────────────────────

OPENING:
"RAG solves the cold-start problem. Instead of fine-tuning (weeks, millions),
you retrieve context at query time. Answers are grounded in actual sources."

KEY STATS:
  - Fine-tuning: 4 weeks, $1M, needs retraining for new data
  - RAG: Immediate, $0.01/query, updatable without retraining
  - Accuracy: +32% (60% without → 92% with grounding)

COMMON QUESTIONS:
  Q: Why not fine-tuning?
  A: Speed (weeks vs. hours), cost ($1M vs. $0), updatability

  Q: How do you prevent hallucination?
  A: Low temperature (0.3), validation layer, explicit instructions

  Q: How would you scale to 10M documents?
  A: Vector DBs handle it, embedding cost is ~$100, query cost negligible

  Q: What's the latency breakdown?
  A: Embed 500ms, search 150ms, LLM 2000ms, total <4s

COMMON MISTAKES
────────────────────────────────────────────────────────────────────────────
  ✗ Different embedding models for query vs. documents (breaks similarity)
  ✗ Huge chunks (>2000 tokens) that mix multiple topics
  ✗ No overlap between chunks (split important info)
  ✗ High temperature with RAG (LLM invents instead of ground)
  ✗ No similarity threshold (accepts low-confidence matches)
  ✗ Not tracking which sources were retrieved (unauditable)
